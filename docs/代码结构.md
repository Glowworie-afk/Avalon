# 📱 前端 Client 目录结构解析

`client` 文件夹是一个高度工程化的 **Taro (React) 小程序**。它完美对应了全栈架构中的“视图层”。

为了方便快速查阅，我们将该目录划分为三个核心维度：**核心业务代码**、**工程化与规范配置**，以及**编译与运行环境**。

## 1. 📂 核心业务代码 (`src` 目录)

这是日常开发最主要的工作区，包含所有的页面、组件和前端逻辑。

* **`src/pages/`**：**页面级组件**。每一个文件夹通常代表小程序的一个独立页面。
  * *例如*：大厅页面存放在 `src/pages/room/index.tsx`。

* **`src/components/`**：**公共/复用组件**。页面中被拆分出来的独立功能模块。
  * *例如*：`<RoleConfig />` (角色配置面板) 和 `<GameBoard />` (游戏主盘)。

* **`src/utils/`**：**纯逻辑工具函数**。存放不涉及 UI 渲染的通用代码。
  * *例如*：`socket.ts` (WebSocket 封装)、`auth.ts` (登录鉴权)。

* **`src/app.ts` & `src/app.config.ts`**：**项目全局入口与配置**。
  * `app.config.ts` 决定了小程序的路由表（Pages）、顶部导航栏颜色、标题等全局表现。

* **`src/app.scss`**：**全局样式文件**。

## 2. 🛡️ 工程化与代码规范 (根目录配置文件)

项目配置了完善的“防呆”机制和代码质量保证工具，规范团队协作。

* **`.husky/` & `commitlint.config.mjs`**：**Git 提交拦截与规范**。
  * 强制要求每次 `git commit` 信息符合格式（如必须带有 `feat:` 或 `fix:` 前缀）。

* **`.eslintrc`**：**JS/TS 语法检查**。
  * 自动发现代码中的低级错误（如变量定义未使用、语法不规范等）。

* **`stylelint.config.mjs`**：**CSS/SCSS 样式检查**。
  * 统一前端样式代码风格。

* **`.editorconfig`**：**编辑器统一个性化**。
  * 确保所有开发者（无论用 VSCode 还是 WebStorm）的缩进（如强制2个空格）和换行符保持一致。

* **`.env.*` (.development / .production / .test)**：**多环境变量**。
  * 区分不同环境的接口地址（如本地测试连 `ws://localhost`，线上连 `wss://api...`）。

## 3. ⚙️ 编译与运行环境

用于将 React/TS 高级语法翻译成微信小程序或浏览器能执行的底层代码。

* **`config/` (dev.ts, prod.ts, index.ts)**：**Taro 框架编译配置**。
  * 区分开发环境（dev）和打包上线环境（prod）的不同构建策略（代码压缩、哈希命名等）。

* **`project.config.json` & `project.private.config.json`**：**微信开发者工具专属配置**。
  * 记录小程序的 AppID 以及开发者工具的本地运行设置。

* **`tsconfig.json`**：**TypeScript 核心配置**。
  * 定义 TS 的编译规则和类型检查严格程度。

* **`babel.config.js` & `.swc/`**：**语法降级工具**。
  * 保证编写的最新的 ES6+ 语法能在老旧型号的手机上顺利运行。

* **`package.json` & `package-lock.json`**：**前端依赖管理**。
  * 记录前端专用的所有第三方依赖包的名称和精确版本锁。

* **`dist/`**：（编译后生成）**最终产物文件夹**。
  * ⚠️ **注意：此文件夹内的代码为打包自动生成，绝对不要手动修改它。**

### 💡 总结与开发建议

作为负责该项目的前端开发者：
1. 你的视线应该 **90% 聚焦在 `src/` 目录**。
2. 根目录下的各类点文件（`.xxx`）和 `config` 文件夹，在项目初期搭建完毕后，**极少需要再次改动**。
3. 新增功能时，按需在 `src/pages` 建页面，在 `src/components` 建组件，通用逻辑尽量往 `src/utils` 里抽离。


# 🧠 服务端 Server 目录结构解析

这是一个非常典型的 **Node.js + TypeScript 后端项目**。它承载了阿瓦隆所有的状态流转、权限校验和游戏核心规则，也就是我们之前强调的 **单一事实来源 (Single Source of Truth)**。

我们可以将这些目录和文件划分为以下四个核心模块：

## 1. 🎮 核心业务与状态层 (大脑与记忆)

* **`game/engine.ts`**：**游戏规则引擎**。
  * 发牌逻辑（洗牌、可见性矩阵计算）、游戏轮次推进、组队提议 (`proposeTeam`)、投票结算 (`castVote`) 全都在这里。它是纯粹的逻辑计算中枢。

* **`store/`**：**数据存储抽象层**。
  * **`roomStore.ts`**：管理所有房间的生命周期（创建、查询、解散）。目前可能是内存 `Map` 实现，未来无缝切换 Redis。房间的状态快照就存在这里。
  * **`userStore.ts`**：管理用户的基本信息（昵称、头像）。

* **`lib/roomCode.ts`**：**业务工具**。
  * 负责生成 4-6 位不重复的房间邀请码（防碰撞逻辑写在这里）。

## 2. 🚪 网络通信层 (大门与信道)

负责和前端（Client）进行数据交换。你的项目同时包含了 HTTP 接口和 WebSocket 长连接。

* **`index.ts`**：**服务主入口**。
  * 这是后端的起步点。从你之前的代码看，这里不仅启动了服务，还包含了核心的 WebSocket 消息分发（`handleMessage`）和广播逻辑（`broadcastRoom`）。

* **`http/`**：**传统 HTTP 接口**。
  * **`login.ts` & `profile.ts`**：在玩家建立 WebSocket 之前，通常需要先通过 HTTP 请求完成微信登录、获取 Token 并更新用户头像/昵称。这些属于“无状态”的短连接请求。

* **`ws/`**：**WebSocket 专属模块**。
  * 目前可能是空的或者做一些 ws 的辅助配置。随着项目变大，你 `index.ts` 里庞大的 `switch-case` 消息处理逻辑，未来可以拆分到这个文件夹里。

## 3. 🛡️ 安全与守卫 (保安)

“永远不要相信客户端传来的数据”，这层专门负责防黑客和权限校验。

* **`middleware/auth.ts`**：**HTTP 鉴权中间件**。
  * 拦截非法的 HTTP 请求，确保只有携带合法凭证的用户才能调用 API。

* **`utils/token.ts`**：**Token 解析工具**。
  * 负责生成和校验 JWT (JSON Web Token)。前端连 WebSocket 时带的 token 就是靠它来解析出真实玩家身份（`openid`）的。

## 4. ⚙️ 基建与测试配置 (机房)

保证服务端能够安全运行、方便调试的底层配置。

* **`config.ts`**：**全局配置中心**。
  * 负责读取环境变量，并提供给其他模块使用（比如运行端口、Redis 地址等）。

* **`.env` & `.env.example`**：**环境变量文件**。
  * 存放极其机密的配置（如数据库密码、微信小程序的 AppSecret）。`.env` 绝对不能提交到 Git（所以有个 `.gitignore` 把他忽略了），而 `.env.example` 只是个空壳占位符，给其他开发者参考的。

* **`test/`**：**单元测试目录**。
  * 这里存放着你用 Jest/Vitest 写的测试文件。之前咱们讨论的“可见性矩阵测试（测梅林能不能看到莫德雷德）”，代码就应该放在这里面。

* **`package.json` & `tsconfig.json`**：**后端专属依赖与 TS 配置**。
  * 这里只安装服务端需要的包，比如 `ws` (WebSocket库)、数据校验库等。不包含任何 React/Taro 相关的包。

* **`_wstest...bak`**：**废弃的测试脚本**。
  * 这些带有 `bak` 和 `del` 后缀的文件，通常是 Vibe Coding (AI) 在帮你尝试写代码、跑测试时留下的“草稿纸”或者备份。确认没用的话可以直接删掉，保持目录整洁。

### 💡 Vibe Coding 开发建议

针对服务端开发，请牢记：

1. **你的主要战场是 `game/` 和 `index.ts`**：阿瓦隆的新机制（如湖中仙女）加在 `game/engine.ts` 里；新事件（如 `ClientEvent.USE_LADY`）加在 `index.ts` 的 `switch` 里。
2. **状态都在 `store` 里**：如果你发现需要存什么长期数据，去改 `store`，千万不要在 `index.ts` 里随手定义一个全局变量（`let xxx = []`），否则一旦服务重启数据就全丢了。

## Day 1 — 房间存储抽象（先 Map，Redis 留口子）

- 写 `store/roomStore.ts`，对外只暴露 `getRoom(id)` / `saveRoom(room)` / `deleteRoom(id)` 三个函数。
- 内部先用内存 `Map<string, Room>` 实现，**别急着上 Redis**（理由见文末选型说明）。
- 接口设计成 async（返回 Promise），将来换 Redis 上层零改动。
- 验证：单测写入/读出/删除一个 Room。

## Day 2 — 房间数据落地 + 建房/入房

- 用 `@avalon/shared` 里的 `Room`/`Player` 类型，房号生成器（4–6 位避重）。
- WebSocket 加 `CREATE_ROOM`/`JOIN_ROOM`/`TOGGLE_READY`/`LEAVE` 处理，改完房间后 `saveRoom` 再 `toPublicRoom()` 广播 `ROOM_UPDATE`。
- 前端首页接通真实建房/入房，大厅显示真实玩家列表（已有 ROOM_UPDATE 骨架，换成真实数据）。
- 验证：两个账号进同一房间、实时互相看到、准备状态同步。

## Day 3 — 房主控制 + 边界

- 房主才能开始游戏、人数校验（5–10）、重复加入/满员/房间不存在拦截。
- 房主退出 → 转让或解散。
- 验证：各种非法操作都被服务端挡住。

## Day 4 — 发牌核心逻辑（先写单测）

全项目逻辑最密集处，**先测后写**。

- `core/roles.ts`：5–10 人角色配置表 + 可见性矩阵（梅林看坏人、派西维尔看梅林/莫甘娜、坏人互认、莫德雷德/奥伯伦例外）。
- `core/dealer.ts`：按人数与角色开关洗牌发牌，给每人算一份 `RoleInfo`。
- 验证：单测覆盖各人数/各扩展组合，重点核对可见性。

## Day 5 — 发牌接口与私密下发

- `START_GAME`：房主开局，服务端发牌、写回 Room。
- **关键安全点**：身份只通过 `ROLE_INFO` 单独发给本人，绝不进任何广播；`ROOM_UPDATE` 永远只发 `PublicRoom`。
- 验证：A 收不到 B 的身份；抓 WebSocket 报文看不到别人身份。

## Day 6 — 身份页接通

- 身份页 mock 换真实 `ROLE_INFO`：长按揭示、松手隐藏（交互已有）。
- 展示夜晚信息。重连后仍能看回自己的身份。
- 验证：7 人局真实发牌，每人身份与夜晚信息正确。

## Day 7 — 对局流程与任务轨道

- 状态机 `core/stateMachine.ts`：五轮任务、组队、投票、否决轨、连续 5 否决坏人胜。
- 投票先藏后揭：服务端收齐再用 `VOTE_RESULT` 公布；`votes` 绝不进 `PublicRoom`。
- 对局页任务轨道接真实数据。
- 验证：完整走完一局五轮。

## Day 8 — 仙女 + 兰斯洛特扩展

- **湖中仙女**：令牌传递、`ladyHistory` 标记「已持有者不可再查」、查验私密展示。
- **兰斯洛特（变体一）**：7 张忠诚卡，第 3–5 轮公开翻牌；翻到「阵营互换」只在本人身份页静默更新阵营，胜负按终局阵营算。
- 联动：仙女查「当前阵营」，兰斯洛特换边后结果跟着变。
- 验证：10 人局开两扩展，换边后查验正确。

## Day 9 — 断线重连 + 边界

- 客户端 `socket.ts` 补自动重连（指数退避，已留口子）；服务端处理掉线、房主迁移。
- 中途退出/非法操作拦截，完整压一局各种异常。
- 验证：真机断网能恢复对局。

## Day 10 — 部署 + 打磨 + 提审

- 后端部署到 https/wss 域名（个人项目可用微信云托管免备案）；小程序后台配置 request/socket 合法域名；`BASE_URL`/`WS_BASE` 按环境切换。
- 体验版发布收 bug、异常文案、隐私协议、类目（「工具」更易过），提交审核。
- 这天是 buffer，别排满。

## 选型说明：房间状态先用 Map，Redis 留作可选

Redis 适合存房间状态的理由：临时数据天生匹配、TTL 自动过期回收死房间、读写快、将来多实例可用 Pub/Sub 跨实例广播。

**但 10 天单人单服务器的 MVP，先用内存 `Map<roomId, Room>` 就够了**，比 Redis 还省事（免装、免连接管理）。Redis 真正值钱的场景是「服务器重启不丢对局」和「多实例」，这两个 10 天内大概率碰不到。

所以建议：Day 1 用 Map 跑起来，把读写封装成 `getRoom()`/`saveRoom()`，内部用 Map 还是 Redis 对上层透明，将来换很轻松。

并发小坑：多人同时操作有「读-改-写」竞争，但 Node 单线程 + 状态放内存 + 消息按到达顺序处理，天然规避；将来换 Redis 时再用乐观锁（WATCH/MULTI）。
